Core Data Models (Contracts)

TypeScript interfaces defining the workspace state machine, O(1) in-memory maps, explicit cryptographic hashing specs, and flat storage payloads.

// O(1) Lookups, Immutability & Formal Transactions
type NodeId = string;
type FileHash = string;

// Explicit Closed State Machine
type WorkspaceState = 'IDLE' | 'AI_PENDING' | 'DIFF_REVIEW' | 'COMMITTING' | 'CONFLICT' | 'ERROR';

interface Workspace {
  id: string;
  version: string;       // Content-addressed: explicitly the rootTreeHash
  state: WorkspaceState; // Current lock/mutation state
  root: DirectoryNode;
  locked: boolean;       // Multi-tab write-lock ownership
}

interface DirectoryNode {
  type: 'dir';
  id: NodeId;
  name: string;
  children: Map<string, FileNode | DirectoryNode>;
}

// Content Storage: Auto-deduplication via blobId = SHA256(content)
interface BlobStore {
  put(content: string | ArrayBuffer): { blobId: string, hash: string };
  get(blobId: string): Promise<ArrayBuffer | string>;
  exists(hash: string): boolean;
}

// Immutable: Content lives separately in the BlobStore
interface FileNode {
  type: 'file';
  id: NodeId;
  name: string;
  hash: FileHash;        // Identity: SHA256("FILE|" + blobId)
  blobId: string;        // Pointer to content, strictly blobId = SHA256(content)
  binary: boolean;       // If >2MB: store blobId only, stream hash, show 'Large File View'
}

// Formal Transaction Lifecycle
interface FileSystemTransaction {
  beginTransaction(): string;
  applyPatch(txId: string, patch: FilePatch): void;
  validate(txId: string): boolean;
  commit(txId: string): void;   // Sets new version = rootTreeHash
  rollback(txId: string): void;
}
// Strict Patch Schemas for LLM Outputs
interface AIEditIntent {
  prompt: string;
  targetFiles: string[]; // Target paths
}

interface FilePatch {
  path: string;
  operations: Array<{
    type: 'replace' | 'insert' | 'delete';
    startLine?: number;
    endLine?: number;
    content: string;
  }>;
}
// Flat Persistence & Formal Merkle Specifications
// SNAPSHOT POLICY:
// 1. Ephemeral (Memory): Max 20 for AI undos. Volatile.
// 2. Commit (Persisted): Bound to IDB transaction history.
interface Snapshot {
  id: string;
  timestamp: number;
  
  // Formal Merkle Hash Definition:
  // dirHash = SHA256("DIR|" + sorted(childName + "|" + childHash).join("|"))
  // fileHash = SHA256("FILE|" + blobId)
  // Recalculation is strictly O(depth) propagating bottom-up.
  rootTreeHash: string; 
  
  // In-memory structural sharing pointer
  tree: DirectoryNode; 
}

// IDB PERSISTENCE (Tier 2): Flat paths + Separate Blob Store
interface IDBPersistenceFormat {
  workspaceId: string;
  version: string;      // Content-addressed rootTreeHash
  
  // 1. Structural Map
  files: Record<string, { // Keyed by absolute path e.g., "src/app.js"
    hash: string;
    blobId: string;
  }>;
  
  // 2. Deduplicated Content Store
  blobs: Record<string, ArrayBuffer | string>; // Keyed by blobId
}

Key Architectural Decisions (ADRs)

The core principles governing system implementation, preventing systemic tech debt and handling edge cases safely.

1

Hash-as-Identity & Immutable Nodes

FileNodes do not store content or dirty flags. They store hash and blobId. By formally defining blobId = SHA256(content), the architecture achieves automatic, zero-cost block-level deduplication across the workspace.

2

O(Depth) Structural Sharing & Hashing

Shadow Trees utilize an explicit path-copying algorithm. When mutating a file, it clones *only* the affected path from leaf to root (O(depth)) while reusing sibling pointers. Hash recalculation identically propagates bottom-up O(depth), preventing catastrophic O(n) full-tree rehashes.

3

No direct UI-to-Disk writes

Monaco onDidChangeContent strictly maps to a separate in-memory blob queue. FileSystemService manages a debounced sync to IndexedDB. Justification: Prevents UI thread blocking during rapid typing.

4

Deterministic Diff Engine

The Diff Engine is designed to transition from naive string-based patching to AST-aware, semantic-safe replacement (Phase 2). Justification: LLM hallucinations or off-by-one line generation errors are safely caught structurally before corruption occurs.

5

Large File Handling Policy

Files larger than 2MB are flagged as binary=true. They bypass Monaco text rendering, store only the blobId, compute hashes via streaming, and display a "Large File View". Justification: Prevents catastrophic browser freezing when parsing massive datasets or bundles.

6

Remote Sync Conflict Strategy

If remote.version != local.version, the state machine enters the explicit CONFLICT state. It suspends standard mutations, pulls the remote snapshot, runs a structural tree diff, and presents a merge overlay. This strictly prevents the silent data loss inherent in Last-Write-Wins.
