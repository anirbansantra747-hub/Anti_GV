📁 MODULE 1 — File System LEFT PANEL

How files and folders are created, stored, edited, deleted — and how they survive a browser refresh. Three storage layers work together: memory → browser → server.

Storage Architecture — 3 Layers
⚡
LAYER 1 — memfs
In-memory file system
POSIX-compatible
All read/writes go here
Lost on page refresh
~0ms latency
⟷
auto-sync
every 3s
💾
LAYER 2 — IndexedDB
localForage wraps IndexedDB
Survives page refresh
Works 100% offline
Browser-only, 50MB+
~5ms latency
⟷
on Ctrl+S
or blur
☁️
LAYER 3 — MongoDB
Workspace collection
Cross-device sync
Source of truth
Files embedded in doc
~50ms latency
💡Rule: Every user action (type, create, delete) hits memfs instantly. IndexedDB captures it automatically for offline-safety. MongoDB only gets called when the user explicitly saves (Ctrl+S) or on window blur — never on every keystroke.

✏️ MODULE 2 — Monaco Editor CENTER PANEL

Monaco is the same engine that powers VS Code. It gives you syntax highlighting, error squiggles, IntelliSense, diff viewer — all for free. Here's how every piece connects to your file system.

Monaco Internal Architecture
📝 EDITOR MODEL
• Each open file = one Monaco Model
• Model backed by memfs content
• Tracks dirty state (unsaved changes)
• Full undo/redo history per file
• Cursor pos saved per tab
🎨 LANGUAGE SERVICES
• Built-in: JS, TS, HTML, CSS, JSON
• ESLint Web Worker → JS errors
• Pyright WASM → Python type errors
• Auto-detects language from extension
• Squiggles = inline error markers
🔍 AI INLINE FEATURES
• Ghost text (inline suggestions)
• Diff editor (accept/reject AI patches)
• Decorations API for AI highlights
• Action menu (right-click AI commands)
• Read selection → send to AI panel
📂 Opening a File in Monaco
1
User clicks file in left panel
react-arborist fires onSelect. File path sent to editor state in Zustand.
2
Content read from memfs
memfs.readFileSync(path, 'utf8'). Returns file content string. If file doesn't exist in memfs → read from IndexedDB first.
memfs.readFileSync
3
Monaco Model created or reused
monaco.editor.createModel(content, language, uri). Uri = file path. If model already exists for this file (open tab), reuse it — preserves cursor + undo history.
createModel()
4
Language mode activates
Monaco detects .java → Java syntax. ESLint worker activates for .js/.ts. Pyright activates for .py. Syntax colors appear instantly.
⌨️ Editing + Error Detection
1
User types code
Monaco fires onChange event. Content written to memfs every keystroke. Tab gets "dirty dot" indicator.
2
Language worker detects errors
ESLint Web Worker runs in background thread. Sends error markers back via postMessage. Monaco.editor.setModelMarkers() draws red squiggles. Hover shows error message. Problems tab shows all errors.

Debounced 3-second timer writes memfs state to localForage. Completely silent. User never sees this. Works even offline.

Explicit save triggers PATCH /api/workspaces/:id. Dirty dot cleared. "Saved" toast shown. File contents in MongoDB updated.
MongoDB sync

🐳 Judge0 Full Flow (Java/C++/Go/C)
1
Client reads file from memfs + sends to backend
POST /api/execute { code, language_id, stdin, time_limit: 10s, memory_limit: 256MB }. JWT header attached.
POST /api/executeJWT auth
2
Backend submits to Judge0 CE
Judge0 adapter calls POST /submissions?wait=false on Judge0 (Railway Docker). Returns { token: "abc123" }. Backend stores token keyed to userId + Socket room.
Judge0 REST API
3
Judge0 compiles + runs in isolated Docker
ephemeral Docker container. seccomp sandbox. No network access. Resource limits enforced by cgroups. Auto-destroyed after job.
Docker isolatedseccomp sandbox
4
Backend polls Judge0 every 500ms
GET /submissions/{token}. Status: Processing → Accepted / Error / Time Limit / Memory Limit. When done, emits result via Socket.io.
5
Result streams to terminal via WebSocket
io.to(userId).emit('execution:result', { stdout, stderr, exitCode, time, memory }). Terminal panel renders: stdout green, stderr red, exit code shown.
Socket.io emitTerminal render
⚠️Error display in editor: When execution produces a runtime error with line numbers (e.g., Java NullPointerException at line 14), the backend parses the stderr, extracts line/column info, and sends back an array of markers. Client calls monaco.editor.setModelMarkers() to draw error squiggles on exact lines — same as VS Code.
🤖 MODULE 4 — AI Panel RIGHT PANEL

Like Cursor AI — the assistant sees your whole workspace, understands your code, and can write patches directly into your files. User sees the diff and accepts or rejects. All using Groq (free, fast).

Context Assembly — What the AI Sees
① ACTIVE FILE
100% of the currently open file content. Always included first. Most important context for the current task.
② OTHER OPEN TABS
First 200 lines of each open tab. Trimmed to keep total context under 6,000 tokens. Gives cross-file awareness.
③ FILE TREE + METADATA
Full directory structure as a string. File names, folder names, language hints. AI knows the project layout.
Active File (full)
+
Open Tabs (200 lines each)
+
File Tree structure
+
User message
+
6-turn history
→
Groq API (Llama 3.3 70B)
⬡ Complete AI Request → Patch Flow
1
User types in AI panel and hits Enter
Examples: "Explain this function", "Fix the bug on line 12", "Write unit tests", "Refactor to async/await". Optional: user highlights code first → selection auto-included in context.
2
Client builds context object
Reads active file from memfs. Reads open tabs from memfs. Gets file tree listing. Estimates total tokens. Trims if over 6,000 tokens (shortest files cut first).
const ctx = {
  activeFile: memfs.readFileSync(activePath, 'utf8'),
  openFiles: openTabs.map(p => trim200(memfs.readFileSync(p))),
  tree: memfs.listAll(), selection: editor.getSelection()
}
3
POST /api/agent/chat → backend
{ message, context, history (last 6 turns), workspaceId }. JWT authenticated. Socket.io room used for streaming response back.
4
Backend builds Groq prompt
System prompt forces JSON response: { type: "explanation" | "patch" | "tests", explanation: "...", patch?: { filePath, newContent } }. Groq Llama 3.3 70B called with stream: true.
Groq APIstream:trueLlama 3.3 70B
5
Tokens streamed back to client via Socket.io
Each token from Groq → io.to(userId).emit('agent:token', token). AI panel shows text appearing in real-time. User sees response building live. When stream ends, emit 'agent:done' with full parsed JSON.
agent:token eventagent:done event
6
If type === "patch" → open Monaco Diff Editor
monaco.editor.createDiffEditor() opens. Left side = original (from memfs). Right side = AI suggested code. User sees exact changes highlighted. Two buttons appear: ✅ Accept / ❌ Reject.
Monaco DiffEditor
7
Accept → write to memfs → autosave triggers
memfs.writeFileSync(patch.filePath, patch.newContent). Monaco model updates live. Dirty dot appears. Undo (Ctrl+Z) fully supported — Monaco tracks this in history. 3-second autosave to IndexedDB kicks in.

📚 Complete Learning Flow — Step by Step
1
User opens Learning Mode from workspace
Slides open as a right-side panel or modal overlay (doesn't close the editor). User can see their code in the background. They type: "I want to learn WebSockets because I'm using them in this project in 2 days."
2
Context sent to backend → Groq asks questions
User message + current workspace file tree + active file → POST /api/learn/start. Groq generates 3–4 clarifying questions: "What language? How much time do you have? Do you want to learn theory or jump to code first? What will you build with this?"
Groq APIworkspace context injected
3
User answers questions (can modify/skip)
Answers are card-by-card, not a boring form. User can type free text or pick options. Answers: "JavaScript, 3 hours, code-first, building a real-time chat app."
4
POST /api/learn/roadmap → Groq generates roadmap JSON
System prompt forces structured JSON. Returns a roadmap with 3–6 steps. Each step has: title, description, type (read/code/exercise), content, estimatedMinutes, starterCode, testCases. Validated with Zod. Saved to MongoDB roadmaps collection.
// Roadmap JSON structure

Roadmap renders as a visual stepper
Steps shown as cards in the learning panel. Lock icon on future steps. Progress bar at top. User clicks "Start Step 1."
6
Exercise steps open in split view in Monaco
Instructions on left, Monaco editor with starterCode on right. User writes code. Clicks ▶ Run. Test cases run via execution engine (same Judge0/Pyodide pipeline from Module 3). Pass/fail shown per test case. User can ask AI for help without leaving this screen.
Monaco editorJudge0 / PyodideAI help inline
7
User ticks ✓ complete → progress saved to MongoDB
PATCH /api/learn/progress { exerciseId, status: 'completed', code: userSolution }. Next step unlocks. Dashboard shows % complete. User can resume later from exact same step.
MongoDB progress
8
"Apply to Project" step — AI writes code into your files
Final step type is "apply." AI generates implementation code based on what user just learned + their project's existing files. Opens Monaco DiffEditor showing what will be added to e.g. server.js. User reviews and clicks Accept — same as Module 4 patch flow. AI can also add a comment in the file explaining what it added.
